Conceptos

Los conceptos son los elementos básicos para construir aplicaciones REST.

El uso correcto de los cuatro conceptos que se van a presentar, junto con las propiedades del protocolo HTTP, hacen que nuestras aplicaciones se ajusten al estilo REST.

Conceptos

Recursos

Un recurso es una pieza de información.

Un recurso es cualquier elemento que tenga suficiente entidad como para que pueda ser referenciado, es decir, para que pueda ser enlazado mediante un link.

Algunos ejemplos:

  • La página de bienvenida de mi empresa.
  • La lista de todos los contactos de mi empresa.
  • Una foto de mis vacaciones.
  • Un vídeo.
  • Una calculadora de la cuota mensual de una hipoteca.

Conceptos

URIs

La URI es el mecanismo que hacen visible mi recurso en la web.

Una URI es como un apuntador a un recurso.

Puede ocurrir que el mismo recurso tenga más de una URI que lo apunta. Pero no puede ocurrir que una URI apunte a más de un único recurso.

Las URIs deben ser lo más descriptivas posible:

  • http://www.uji.es/agenda/entradas/ Es claro.
  • http://www.uji.es/1/as2/ Es críptico.

Conceptos

Representación

Un mismo recurso puede tener más de una representación.

La lista de todas las entradas de mi agenda es un recurso. Este recurso lo puedo representar en formato XML, como una hoja de cálculo, como una tabla, o incluso como un pdf. En todos los casos el recurso es el mismo, lo que cambia es su representación.

Conceptos

Enlaces

Una de las características que han hecho de la Web una tecnología de éxito es el hecho de que es posible establecer enlaces entre distintos recursos.

En una página web podemos incluir un enlace a otra página web, o a una imagen, o a cualquier otro recurso accesible en la web.

Es importante que la representación de mis recursos contengan enlaces a otros recursos.

Al fin y al cabo, este es el significado de Hypertext.

Propiedades

Las propiedades del protocolo HTTP han hecho de él una tecnología de éxito.

Estas propiedades son sencilla, pero es fácil que las olvidemos cuando desarrollamos nuestras aplicaciones REST.

Propiedades

Direccionable

Como hemos visto, los recursos y las URIs que apuntan hacia ellos son conceptos fundamentales en la Web (y en las aplicaciones REST).

Una aplicación REST debe exponer todos los recursos de interés a través de URIs.

Este es el significado de direccionable, los recursos de mi aplicación son accesibles a través de las URIs que he definido para ello.

Propiedades

Sin estado

HTTP es un protocolo sin estado.

Que HTTP no tenga estado implica que, un servidor no tiene memoria (estado) entre dos peticiones sucesivas. Cada petición es vista como completa e independiente de cualquier otra.

Que HTTP sea un protocolo sin estado no significa que nuestras aplicaciones no tengan estado. De hecho, REST significa Representational State Transfer, el estado de la aplicación (su representación) se transfiere entre el cliente y el servidor para dar lugar a aplicaciones con estado.

En aplicaciones web con estado es común hacer uso de Cookies para almacenar el estado de la interacción. En las aplicaciones REST las Cookies están totalmente desaconsejadas.

Propiedades

Conexión

Esta propiedad está directamente relacionada con el concepto de Direccionabilidad.

Uno de los factores de éxito de la web es su navegabilidad, los recursos expuestos en la web se pueden navegar gracias a los enlaces que existen entre ellos.

La web es un conjunto de recursos conectados, navegables.

Propiedades

Interfaz uniforme

Cuando se presentaron los métodos disponibles en el protocolo HTTP, vimos que todos ellos están disponibles para cualquier recurso.

Dicho de otro modo, sea cual sea el recurso, siempre disponemos del mismo conjunto de métodos para acceder a cualquier recurso.

Cada uno de los métodos tiene una semántica bien definida. Nuestras aplicaciones siempre deben seguir la semántica de los métodos HTTP, no podemos reinterpretar el significado de los métodos.

Por ejemplo, GET se usa para leer un recurso. Nuestra aplicación no debería hacer uso de GET para modificar el estado de un recurso.

Resumen

En este capítulo hemos visto cuales son los conceptos y propiedades que debe poseer una aplicación Web para que pueda ser clasificada como RESTful.

Estos conceptos y propiedades, aunque sencillos, son fáciles de obviar.
